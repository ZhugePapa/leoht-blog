## 2. 结构问题诊断

### 2.1 致命问题：全是剪藏，没有知识沉淀

**这是最核心的问题，比目录结构严重得多。**

548 篇文章全部是原文剪藏，没有一篇有你的个人理解、应用场景、相关链接。这意味着：

- 现在的"知识库"其实是"收藏夹"——正好是 product-spec.md §0 想要避免的反面案例
- product-spec.md §3.2 的 frontmatter 要求 `summary`（一句话核心观点）、`type`、`status`、`tags`（多个语义标签）、`related`（关联内容）、`source_*` 字段，现有文章一个都没填
- 现有 `tags: [clippings]` 是 Obsidian Clipper 的默认值，对知识组织毫无意义——548 篇文章共享同一个标签

**结论：这些文章不是"可发布的内容"，是"待加工的原料"。** 它们进系统时 `status` 应该是 `inbox`，需要你逐篇加工后才能变成 `published`。

### 2.2 L1 / L1_sc 双套目录的处理

两套目录结构完全镜像，`L1` 是英文原文，`L1_sc` 是中文翻译。

**建议：以 `L1_sc`（中文）为主，`L1`（英文）降级为参考。**

理由：
- 你的博客面向中文读者，发布用中文
- 英文原文的价值是"溯源"——当你要写"我的理解"时回看原文，但不需要作为独立内容发布
- 双套并存会让内容库看起来有 548 篇，实际有效内容只有 270 篇，统计失真

**落地方式**：
- 迁入时只迁 `L1_sc`，`L1` 保留在原 Obsidian 库里作为参考
- 或迁入时把 `L1` 的文章塞进 `source_url` 字段做溯源，不作为独立 Card

### 2.3 目录嵌套过深，且与标签体系冗余

现有 4 层嵌套：

```
L1 → 1.1 人与行为 → 1. 行为经济学 → loss aversion UX → Loss Aversion.md
```

第 4 层（`loss aversion UX` 这种"主题 + UX"命名）其实是标签，不是分类。product-spec.md §4 已经明确：**MVP 用扁平标签，不用 Domain/Topic 三层分类。**

**建议：砍掉第 3、4 层，把层级降到 2 层，第 3、4 层信息转成标签。**

```
现有：L1/1.1 人与行为/1. 行为经济学/loss aversion UX/Loss Aversion.md
建议：content/behavior/loss-aversion.md
      tags: [行为经济学, 损失厌恶, 决策, 定价, 订阅转化]
```

- 第 1 层 `L1` 删掉（迁入时就是 content 目录）
- 第 2 层 `1.1 人与行为` → `behavior/`（slug 化）
- 第 3 层 `1. 行为经济学` → 标签 `行为经济学`
- 第 4 层 `loss aversion UX` → 标签 `损失厌恶`

这样目录只剩一层（6 个 Domain 文件夹），每篇文章靠标签横向连接。

### 2.4 一级分类只覆盖了 2 个 Domain

product-spec.md §6.2 规划了 6 个 Domain，目前目录只有 `1.1 人与行为`（behavior）和 `1.2 体验设计 UX`（ux）。其余 4 个（growth / product / thinking / case-study）还没内容。

**这不是问题，是事实。** MVP 不强求 6 个 Domain 都有内容，先迁已有的 2 个，其余随写随加。

### 2.5 文件名中英混杂、含特殊字符

`L1` 文件名是英文标题原文，`L1_sc` 是中文翻译标题。问题：
- 中英混杂不利于 URL slug 生成
- 部分文件名含特殊字符（`:`、`?`、`()`、`/`），迁入时需要 slug 化

**建议**：迁入时统一生成 `slug`（kebab-case 英文），文件名用 slug，中文标题放 frontmatter `title` 字段。

## 3. 文章格式与 product-spec.md 的差距

### 3.1 frontmatter 对照

| product-spec.md 字段 | 现有文章 | 差距 |
| --- | --- | --- |
| `title` | ✅ 有 | OK，中文版可直接用 |
| `slug` | ❌ 没有 | 迁入时自动生成 |
| `summary` | ⚠️ 有 `description` 但那是原文摘要，不是"你的核心观点" | 需要你逐篇写 |
| `content` | ✅ 有正文 | OK，但需要加"我的理解"等段落 |
| `type` | ❌ 没有 | 迁入时根据目录推断（如 `effect` / `principle` / `case`） |
| `status` | ❌ 没有 | 迁入时统一设为 `inbox` |
| `tags` | ❌ 只有 `[clippings]` | 需要逐篇打语义标签 |
| `source_title` | ⚠️ 有 `source`（URL）但无标题 | 从原文推断 |
| `source_author` | ⚠️ 有 `author` 但格式是 `[[Sarah Thompson]]` | 需要清洗 |
| `source_url` | ✅ 有 `source` | OK |
| `published_at` | ⚠️ 有 `published`（原文发布日期） | 可用，但要区分"原文发布日"和"你发布日" |
| `related` | ❌ 没有 | 需要逐篇添加 |

### 3.2 正文内容差距

现有正文是纯原文搬运，product-spec.md §6.3 文章页期望的结构是：

```
核心观点（一句话）
正文
原文摘录
我的理解        ← 全部缺失
UX / 产品 / 增长应用  ← 全部缺失
相关卡片        ← 全部缺失
来源
```

**结论：548 篇文章没有一篇满足发布标准。** 它们都需要加工——补"我的理解"和"应用场景"——才能从 `inbox` 升到 `published`。

### 4.2 内容加工（必须手动，无法自动化）

- 冷启动：挑 5-10 篇最重要的，完整加工后作为首批发布内容

## 6. 一个诚实判断

你的知识库现在是一个**高质量的原料库**，不是知识系统。原料好——nngroup 和 The Decision Lab 是 UX 领域最权威的来源——但 548 篇剪藏没有一篇有你的思考，它们的价值等于零，直到你写出"我的理解"那一刻才产生。

这不是批评。大多数人的知识库都是这个状态——收藏多、复用少。但如果你不在项目启动前正视这一点，博客上线后会面临一个尴尬：548 篇文章，0 篇可发布。

所以我的建议是：**别想着把 548 篇都加工完。** 挑 5-10 篇最重要的，认真写出"我的理解"和"应用场景"，作为冷启动内容。其余的留在 `inbox`，以后慢慢加工。inbox 里的 540 篇不是负担，是资产——它们是你未来写文章的原料库，等你需要写某个主题时回去翻。
