版本：v0.1  
日期：2026-06-13  
产品定位：Personal Knowledge System for UX Designers  
文档类型：产品需求文档 / Product Requirement Document

---

## 1. 产品概述

### 1.1 产品一句话

一个面向体验设计师的个人知识系统，用于沉淀 UX、心理学、增长、产品、方法论与案例知识，并通过结构化卡片、标签体系、双向链接和知识图谱，帮助设计师复用认知、支撑设计决策、持续输出内容。

### 1.2 产品定位

本产品不是传统个人博客，也不是单纯收藏夹，而是：

> UX 设计师的第二大脑：Notion + Roam Research + UX Blog 的个人化结合。

它兼具三种角色：

- **内容输出平台**：对外发布文章、案例、方法论和设计思考。
- **知识管理系统**：对内沉淀摘录、卡片、主题、关联概念与复盘。
- **设计决策辅助库**：在实际设计问题中快速检索原则、案例、心理学效应、增长机制和可用性依据。

### 1.3 核心价值

- 将零散输入变成可检索、可复用、可关联的认知资产。
- 帮助体验设计师建立跨领域知识结构。
- 支撑从用户研究、交互设计、增长优化到案例复盘的设计判断。
- 让个人博客从“文章列表”升级为“知识网络”。

## 2. 背景与问题

### 2.1 当前问题

体验设计师日常会接触大量信息：

- 心理学效应
- UX 原则
- 用户研究方法
- 增长模型
- 产品案例
- AI 产品设计经验
- 项目复盘
- 阅读摘录

但这些信息往往存在几个问题：

- 收藏多，复用少。
- 标签多，结构弱。
- 文章彼此孤立，无法形成知识网络。
- 输入和输出断裂，读过的内容很难变成可发表文章。
- 设计决策时找不到过去沉淀的证据、案例和原则。

### 2.2 产品机会

传统博客解决的是“发布”，笔记软件解决的是“记录”，知识图谱工具解决的是“关联”。本产品的机会在于把三者合并为一个面向 UX 设计师的专用系统：

- 用 blog 形态承载公开表达。
- 用知识库模型承载内部沉淀。
- 用 UX 设计知识模型组织内容。
- 用关联网络帮助设计师发现知识之间的关系。

## 3. 目标用户

### 3.1 核心用户

**内容创作者型体验设计师**

- 职业：UX 设计师、产品设计师、交互设计师、体验策略设计师。
- 行为：长期阅读、摘录、写作、复盘和输出。
- 需求：把心理学、UX、增长和案例知识沉淀成个人方法库。

### 3.2 次级用户

**同行设计师 / 产品经理 / 研究者**

- 希望阅读高质量 UX 知识内容。
- 希望通过主题、标签、案例和知识图谱探索相关内容。
- 关注设计方法、产品增长、AI UX 和案例拆解。

### 3.3 未来扩展用户

**设计学生 / 初级设计师**

- 需要系统学习 UX 知识结构。
- 需要从案例和方法论中建立设计判断。

## 4. 产品目标

### 4.1 用户目标

- 快速记录与整理 UX 相关知识。
- 用统一结构沉淀每篇文章、摘录和案例。
- 能按领域、主题、标签、设计阶段和行为层级检索内容。
- 能看到内容之间的关系，形成个人知识网络。
- 能把内部卡片逐步转化为公开文章。

### 4.2 产品目标

- 搭建一个可长期维护的 UX 知识系统。
- 形成清晰的信息架构和内容模型。
- 支持个人创作者的输入、整理、输出闭环。
- 兼顾公开阅读体验和后台管理效率。

### 4.3 非目标

当前阶段不优先做：

- 多用户协作。
- 复杂社区系统。
- 商业课程售卖。
- 全量 AI 自动写作。
- 重型项目管理。

## 5. 产品原则

### 5.1 结构优先于数量

内容多不等于知识系统强。每条内容都应进入明确的 Domain / Topic / Card 结构，并尽量建立关联。

### 5.2 每篇内容都是可复用认知单元

文章不只是阅读材料，也应是未来设计决策、写作、案例分析和项目复盘可以再次调用的卡片。

### 5.3 关联比分类更重要

分类回答“它属于哪里”，关联回答“它还能解释什么”。本产品必须强化双向链接和相关概念。

### 5.4 公开表达与私人沉淀分层

同一个知识对象可以有不同状态：

- 私人摘录
- 草稿卡片
- 完整文章
- 公开发布
- 收藏 / 稍后读

### 5.5 面向设计决策，而不只是阅读

系统应帮助用户回答：

- 这个问题背后是什么用户行为？
- 有哪些心理学或 UX 原则可解释？
- 有没有相关案例？
- 可以如何应用到产品设计或增长优化中？

## 6. 知识结构模型

### 6.1 三层知识模型

```text
Domain 领域
  └── Topic 主题
        └── Card 知识卡片 / 文章 / 摘录
```

### 6.2 Domain 领域

建议第一版设置 6 个领域：

| Domain | 中文名 | 解决的问题 |
| --- | --- | --- |
| Behavior | 人与行为 | 用户为什么这样做 |
| UX | 体验设计 | 产品应该如何设计 |
| Growth | 产品增长 | 设计如何影响业务 |
| Product | 产品系统 | 设计如何落地 |
| Thinking | 方法论与思维 | 设计师如何思考 |
| Case Study | 案例研究 | 别人是如何做的 |

### 6.3 Topic 主题示例

**Behavior / 人与行为**

- 行为经济学
- 决策理论
- 认知科学
- 注意力机制
- 动机理论
- 情绪设计

**UX / 体验设计**

- 交互设计
- 信息架构
- 可用性原则
- 设计系统
- 用户研究
- 可访问性
- 服务设计
- 内容设计 / UX Writing

**Growth / 产品增长**

- AARRR / RARRA
- 激活
- 留存
- 转化率优化
- 习惯养成设计
- 定价心理学
- 北极星指标

**Product / 产品系统**

- 前端理解
- 产品架构
- 数据分析
- A/B 测试
- AI 产品设计
- API / 系统设计理解

**Thinking / 方法论与思维**

- 设计思维
- 系统思考
- 第一性原理
- JTBD
- 双钻模型
- 结构化思考

**Case Study / 案例研究**

- 优秀产品拆解
- 行业趋势
- 设计案例复盘
- 失败案例分析
- 设计访谈总结

## 7. 内容对象模型

### 7.1 核心对象

| 对象 | 说明 |
| --- | --- |
| Domain | 知识大领域 |
| Topic | 领域下的主题 |
| Card | 最小知识单元，可为文章、摘录、原则、案例、效应 |
| Article | 面向公开阅读的长内容 |
| Note | 私人笔记或临时记录 |
| Highlight | 原文摘录 |
| Tag | 横向标签 |
| Concept | 可被图谱连接的概念节点 |
| Link | 内容之间的双向关联 |
| Collection | 收藏夹 / 专题 |
| Source | 书籍、文章、论文、播客、视频等来源 |

### 7.2 Card 内容模板

每个 Card 建议统一包含：

```text
标题
所属 Domain / Topic
核心观点（一句话）
内容类型
原文摘录
我的理解
应用场景（UX / 产品 / 增长）
相关内容链接
标签
来源
状态
创建时间 / 更新时间
```

### 7.3 内容类型

- 长文
- 札记
- 摘录
- 心理学效应
- 设计原则
- 增长机制
- 产品案例
- 项目复盘
- 读书笔记
- 灵感碎片

### 7.4 内容状态

- Inbox：快速记录，未整理
- Draft：草稿中
- Structured：已结构化
- Published：已公开
- Archived：归档
- Deleted：回收站

## 8. 标签系统

标签不替代分类，而是用于横向连接内容。

### 8.1 标签维度

**A. 内容类型标签**

- 心理学
- 案例
- 方法论
- 框架
- 原则
- 数据
- 复盘

**B. 设计阶段标签**

- 研究
- 定义
- 设计
- 验证
- 评估
- 增长

**C. 行为层级标签**

- 认知
- 决策
- 行为
- 情绪
- 注意力
- 动机

**D. 应用场景标签**

- 注册登录
- Onboarding
- 订阅转化
- 信息架构
- 表单设计
- 空状态
- AI 对话
- 数据看板

### 8.2 标签规则

- 一篇内容必须有 1 个 Domain 和 1 个 Topic。
- 一篇内容可以拥有多个标签，建议 3-7 个。
- 标签应优先表达“可复用场景”，避免只表达文章表面主题。
- 标签需要统计使用次数，并支持合并、改名和分组。
- 文章卡片、文章详情页、搜索结果和知识图谱节点上的标签都应可点击。
- 点击标签后进入标签聚合页，快速查看所有拥有同一标签的文章和卡片。

### 8.3 标签浏览机制

标签是读者横向探索知识的主要入口。它不只用于内容管理，也承担“同类文章快速浏览”的阅读路径。

核心机制：

- **标签入口**：首页热门标签、文章卡片标签、文章详情页标签、搜索结果标签、内容库筛选标签。
- **标签聚合页**：每个标签拥有独立页面，展示该标签下的所有文章、卡片、摘录和案例。
- **标签组合筛选**：在标签页中可继续添加第二个标签，查看交集内容，例如 `#信息架构 + #Onboarding`。
- **标签上下文**：标签页需要说明该标签的定义、所属标签组、相关文章数量、常见关联标签。
- **标签排序**：支持按最新、最热、收藏数、内容类型排序。
- **标签反向发现**：在标签页展示“经常一起出现的标签”，帮助用户发现相关主题。

标签页基础信息：

```text
标签名称
标签描述
标签分组
文章数量
卡片数量
相关 Domain / Topic
关联标签
最新内容
热门内容
```

## 10. 核心页面需求

### 10.1 Home 首页

目标：建立产品定位，展示最新内容，并提供进入知识系统的路径。

核心模块：

- 个人品牌与价值主张
- 最近更新
- 三到六个 Domain 入口
- 今日推荐卡片
- 热门标签
- 最新文章列表
- 订阅入口

首页不应只是文章堆叠，而应体现“UX 知识系统”的结构感。

### 10.2 Knowledge Graph 知识图谱

目标：展示知识之间的关联，是产品差异化核心。

核心能力：

- 节点代表 Card / Concept / Topic。
- 连线代表关联关系。
- 支持按 Domain、Topic、Tag 过滤。
- 点击节点可查看摘要、关联内容和入口链接。
- 支持从一篇文章跳转到局部图谱。

关联示例：

```text
损失厌恶
  → 定价设计
  → 转化优化
  → 行为经济学
```

MVP 阶段可以先做轻量图谱：

- 不做复杂物理引擎。
- 先做“相关节点网络”。
- 每个文章页展示局部关联图。

### 10.3 Library 内容库

目标：提供结构化浏览和筛选。

核心能力：

- 卡片流 / 列表视图切换。
- 按 Domain 筛选。
- 按 Topic 筛选。
- 按内容类型筛选。
- 按标签筛选。
- 按状态筛选。
- 支持收藏、稍后读、批量管理。

### 10.4 Article 文章页

目标：同时服务阅读、理解和关联探索。

推荐布局：

- 左侧：目录 / 所属 Domain / Topic。
- 中间：正文内容、摘录、作者理解、应用场景。
- 右侧：相关卡片、相关概念、局部知识图谱。
- 顶部或正文尾部：展示文章标签，标签可点击进入同标签内容列表。

文章内容结构：

```text
标题
一句话观点
正文
原文摘录
我的理解
UX / 产品 / 增长应用
相关卡片
来源
```

### 10.5 Tag 标签页

目标：让读者通过一个标签快速查看所有同标签文章，并继续发现相关主题。

核心能力：

- 展示标签名称、标签描述、所属标签组和内容数量。
- 聚合所有包含该标签的 Article / Card / Note / Highlight。
- 支持按内容类型筛选，例如长文、案例、方法论、摘录、复盘。
- 支持按 Domain / Topic 二次筛选。
- 支持与其他标签组合筛选，形成交集内容列表。
- 展示相关标签，例如 `#注意力机制` 可关联 `#信息架构`、`#认知负荷`、`#转化设计`。
- 展示该标签下的精选内容、最新内容和高价值内容。

标签页推荐结构：

```text
标签标题
标签解释
内容统计
相关标签
筛选器
文章 / 卡片列表
推荐继续探索的 Topic
```

典型使用场景：

```text
用户在文章中看到 #损失厌恶
  → 点击标签
  → 进入 #损失厌恶 标签页
  → 查看所有相关心理学效应、定价设计、转化优化文章
  → 继续点击 #定价心理学 或 #转化率优化
```

### 10.6 Search 搜索页

目标：让用户快速找回知识。

核心能力：

- 全站搜索。
- 支持关键词、标题、正文、标签、来源搜索。
- 结果按文章、卡片、标签、专题分组。
- 支持过滤：Domain / Topic / 类型 / 时间 / 状态。
- 展示搜索历史和热门搜索。

### 10.7 Admin 内容管理

目标：让创作者高效输入、整理、发布。

核心能力：

- 内容库管理。
- 新建 Card / Article / Note。
- Markdown 编辑器。
- 草稿箱。
- 批量移动、删除、打标签。
- 回收站。
- 关联内容推荐。
- 导出备份。

### 10.8 About 关于页

目标：建立个人可信度和知识系统背景。

核心模块：

- 个人简介。
- 设计理念。
- 经历与项目。
- 推荐书单 / 资源。
- 联系方式。

### 10.9 Subscribe 订阅页

目标：建立长期读者关系。

核心能力：

- 邮件订阅。
- 社交媒体链接。
- 关注入口。
- 通知设置。

## 11. 核心用户流程

### 11.1 输入到发布流程

```text
快速记录
  → 进入 Inbox
  → 补充 Domain / Topic / Tags
  → 添加核心观点与来源
  → 建立相关链接
  → 扩写为文章
  → 发布
```

### 11.2 阅读到探索流程

```text
进入文章
  → 阅读核心观点
  → 点击文章标签
  → 查看同标签文章
  → 查看相关卡片
  → 打开局部知识图谱
  → 进入相关 Topic
  → 收藏 / 订阅
```

### 11.3 设计决策检索流程

```text
输入设计问题
  → 搜索关键词
  → 筛选 Domain / Topic
  → 查看相关原则与案例
  → 打开应用场景
  → 复用到设计方案
```

### 11.4 标签探索流程

```text
看到感兴趣的标签
  → 点击标签
  → 进入标签聚合页
  → 浏览同标签文章 / 卡片 / 摘录
  → 使用 Domain / Topic / 类型筛选
  → 点击相关标签继续探索
  → 收藏或进入文章详情
```

## 14. 内容策略

### 14.1 内容生产节奏

建议采用三层输出：

- 每周：短卡片 / 摘录 / 灵感碎片。
- 每月：长文 / 方法论文章 / 案例拆解。
- 每季度：专题复盘 / 知识图谱整理。

### 14.2 内容质量标准

每篇公开内容至少满足：

- 有明确观点。
- 有来源或案例支撑。
- 有个人理解，而不是搬运。
- 有 UX / 产品 / 增长应用场景。
- 至少关联 2 个已有内容。

### 14.3 推荐专题

- 心理学如何影响产品设计
- UX 设计师的增长知识
- AI 产品体验设计
- 设计系统与一致性
- 从案例学习信息架构
- 设计决策中的数据与实验

## 15. 设计体验要求

### 15.1 信息体验

- 页面应强调结构和可探索性。
- 不用传统博客时间线作为唯一入口。
- 分类、标签、图谱和推荐内容都应帮助读者继续探索。

### 15.2 阅读体验

- 长文排版应舒适、克制、低干扰。
- 正文宽度控制在 680-760px。
- 右侧关联内容不应抢正文注意力。
- 代码、摘录、引用、方法框架应有明显但克制的区分。

### 15.3 管理体验

- 新建内容要快。
- 补充结构信息要轻。
- 标签和关联不应变成负担。
- 系统应鼓励“先记录，再整理”，避免一开始就要求完整填写。

## 16. 成功指标

### 16.1 创作者侧指标

- 每周新增 Card 数。
- 每月发布文章数。
- 已结构化内容占比。
- 平均每篇内容关联数。
- 标签复用率。
- 从 Inbox 到 Published 的转化率。

### 16.2 读者侧指标

- 平均阅读时长。
- 相关文章点击率。
- 搜索使用率。
- 标签页访问率。
- 收藏 / 订阅转化率。
- 回访率。

### 16.3 知识系统健康度

- 孤立内容比例。
- 无标签内容比例。
- 无 Domain / Topic 内容比例。
- 高频主题增长趋势。
- 被引用最多的概念节点。

第一版应该先做“可持续使用”的最小闭环：

- 读者可以搜索到目标内容。
- 系统能体现“知识网络”而非普通博客列表。

## 19. 风险与对策

| 风险 | 表现 | 对策 |
| --- | --- | --- |
| 分类过重 | 每次记录都要填很多字段 | 允许 Inbox 快速记录，后整理 |
| 标签失控 | 标签越来越多但不可用 | 设置标签分组、合并和使用统计 |
| 图谱噱头化 | 好看但不实用 | 先做局部相关图谱，服务文章阅读 |
| 内容发布压力大 | 只想发布完整长文 | 允许卡片、摘录、札记等轻内容 |
| 读者不知道怎么逛 | 首页只有文章列表 | 提供 Domain、Topic、热门标签、推荐卡片 |

## 20. 下一步建议

建议下一步产出三份更细文档：

1. **信息架构文档**：明确所有页面、导航、对象关系和状态流转。

```text

  → 最后进入技术实现
```
