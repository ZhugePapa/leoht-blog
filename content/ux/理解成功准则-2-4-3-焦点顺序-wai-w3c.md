## 理解焦点顺序 (级别 A)

## 成功准则 (SC)

## 意图

此成功准则的意图是确保当用户顺序导航内容时，他们遇到的信息顺序与其内容的含义一致，并且可以通过键盘进行操作。这通过让用户对内容形成一致的心理模型来减少困惑。内容中可能存在反映逻辑关系的不同顺序。例如，在表格的组件中一次移动一行或一次移动一列都反映了内容中的逻辑关系。这两种顺序都可以满足此成功准则。

在网页内容中确定顺序导航顺序的方式是由内容的底层技术决定的。例如，简单的 HTML 通过制表符顺序（tabbing order）的概念来定义顺序导航。动态 HTML 可以使用脚本结合添加 `tabindex` 属性来修改导航序列，以允许焦点落在其他元素上。如果不使用脚本或 `tabindex` 属性，则导航顺序即为组件在内容流中出现的顺序。（参见 [HTML 现行标准中的焦点](https://html.spec.whatwg.org/multipage/interaction.html#focus)）。

不属于此成功准则所针对的顺序导航的键盘导航示例是使用方向键导航来遍历树形组件。用户可以使用向上和向下方向键在树节点之间移动。按向右方向键可以展开节点，然后使用向下方向键将移入新展开的节点中。此导航序列遵循树形控件的预期序列——随着其他项的展开或折叠，它们会被添加或从导航序列中移除。

只要用户仍能理解和操作网页，焦点顺序就不需要与通过程序确定的阅读顺序完全相同（参见 [成功准则 1.3.2 有意义的序列](https://www.w3.org/WAI/WCAG22/Understanding/meaningful-sequence.html)）。由于内容可能有几种可能的逻辑阅读顺序，因此焦点顺序可以与其中任何一种相匹配。然而，当特定呈现的顺序与通过程序确定的阅读顺序不同时，使用其中一种呈现的用户可能会发现难以理解或操作网页。作者在设计网页时应仔细考虑所有这些用户。

例如，屏幕阅读器用户与通过程序确定的阅读顺序进行交互，而有视力的键盘用户与网页的视觉呈现进行交互。应注意使焦点顺序对这两类用户都有意义，并且对他们中的任何一方都不会显得是在随机跳跃。

焦点顺序需要允许用户以逻辑顺序导航可聚焦元素，并且该顺序需要保留页面所传达的任何含义或操作。焦点顺序不一定需要遵循网页的视觉布局，只要元素接收焦点的顺序是逻辑性的，并且保留了视觉呈现所隐含的内容层级 and 关系即可。例如，在双栏布局中，西方从左到右系统的假设可能是焦点首先移过左栏中的元素（从该栏的顶部到底部），然后继续移到右栏。然而，如果这两栏相互独立，并且含义/操作不受影响，那么如果右栏中的元素首先接收焦点，随后是左栏中的元素，这并不算是不符合。不过，作为最佳实践，请确保焦点顺序强化了视觉布局所隐含的阅读顺序。

注意

该要求没有规定什么应该或不应该接收焦点，而是评估元素接收焦点的顺序。此成功准则并不禁止使不可操作的内容（例如静态文本）可聚焦，或通过程序将焦点移动到此类内容上。然而，如果项目接收焦点的顺序阻碍了内容的含义或操作，或者创建了令人困惑或不合逻辑的焦点顺序，则是焦点顺序的失败——例如，由于使用嵌套的可聚焦元素，一个控件似乎多次接收焦点。

```html
<div tabindex="0"><button>...</button></div>
```

作为最佳实践，请避免设置无法操作或触发的可聚焦元素，因为这些很可能会让键盘用户的操作变得繁琐。

为了明确起见：

- 可聚焦组件需要以保留含义和可操作性的顺序接收焦点。
- 焦点顺序不一定需要遵循视觉呈现或布局。
- 如果存在多种保留含义和可操作性的顺序，则只需提供其中一种。
- 静态/非交互式元素可以接收焦点，只要它们不阻碍内容的操作，或导致令人困惑或不合逻辑的焦点顺序。

## 收益

这些技术使顺序导航文档并期望焦点顺序与顺序阅读顺序保持一致的键盘用户受益。

- 必须依靠键盘访问来操作页面的行动障碍人士可以从逻辑、可用的焦点顺序中受益。
- 存在阅读障碍的残疾人士在 Tab 键将焦点带到意想不到的地方时可能会感到迷失方向。他们可以从逻辑焦点顺序中受益。
- 视力障碍人士在 Tab 键将焦点带到意想不到的地方，或者当他们无法轻易找到交互式元素周围的内容时，可能会感到迷失方向。
- 对于使用高放大倍率屏幕放大镜的人来说，页面中只有一小部分是可见的。如果焦点顺序不合逻辑，此类用户可能会在错误的上下文中理解字段。

## 示例

- 在包含交互式控件树的网页上，用户可以使用向上和向下方向键在树节点之间移动。按向右方向键展开节点，然后使用向下方向键移入新展开的节点。
- 网页实现非模态对话框。当触发按钮被激活时，对话框打开。对话框中的交互式元素会紧接着插入在按钮之后的焦点顺序中。当对话框打开时，焦点顺序从按钮转到对话框的元素，然后转到紧跟在按钮之后的交互式元素。当对话框关闭时，焦点顺序从按钮转到其后的元素。
- 网页实现模态对话框。当触发按钮被激活时，对话框打开，并且焦点被设置在对话框内。只要对话框处于打开状态，对话框之外的所有网页内容都会变得不可交互且无法接收焦点（不过，根据实现的不同，焦点循环可能仍会包括用户代理控件）。当对话框被关闭时，焦点返回到按钮或按钮之后的元素。
- 一个 HTML 网页在 HTML 中将左侧导航栏放在主体内容之后，并使用 CSS 样式使其显示在页面的左侧。这样做是为了允许焦点首先移至主体内容，而不需要 `tabindex` 属性或 JavaScript。
	注意
	虽然此示例符合成功准则，但并非所有的 CSS 定位都必然符合。更复杂的定位示例可能会也可能不会保留含义和可操作性。
- 以下示例**未能满足成功准则**：
	某家公司的网站包含一个表单，用于收集市场营销数据，并允许用户订阅该公司出版的几份电子报。用于收集市场营销数据的表单部分包括姓名、街道地址、城市、省份和邮政编码等字段。表单的另一个部分包括几个复选框，以便用户可以指明他们想要接收的电子报。然而，表单的 Tab 键顺序在表单的不同部分之间跳跃，使得焦点从姓名栏移动到复选框，然后移到街道地址，再移到另一个复选框。

## 关键术语

辅助技术 (assistive technology)

充当[用户代理](https://www.w3.org/WAI/WCAG22/Understanding/#dfn-user-agent)或与主流用户代理一起运行的硬件和/或软件，以提供超出主流用户代理所能提供的功能，从而满足残疾用户的需求

注意 1

辅助技术提供功能包括替代呈现（例如合成语音或放大内容）、替代输入方法（例如语音）、附加导航或定位机制，以及内容转换（例如使表格更具可访问性）。

注意 2

辅助技术通常通过使用和监控 API 来与主流用户代理传递数据和消息。

注意 3

主流用户代理与辅助技术之间的区分并不是绝对的。许多主流用户代理提供了一些功能来协助残疾人。基本的区别在于主流用户代理针对的是广泛且多样化的受众，这些受众通常包括有残疾和无残疾的人。辅助技术针对的是具有特定残疾的、定义狭窄的用户群体。辅助技术提供的协助更具体且适合其目标用户的需求。主流用户代理可以向辅助技术提供重要功能，例如从程序对象检索网页内容或将标记解析为可识别的包。

键盘接口 (keyboard interface)

软件用于获取按键输入的接口

注意 1

键盘接口允许用户向程序提供按键输入，即使原生技术不包含键盘。

注意 2

通过键盘操作的鼠标模拟器（如 MouseKeys）来操作应用程序（或应用程序的一部分）不符合通过键盘接口操作的条件，因为程序的操作是通过其定位设备接口，而不是通过其键盘接口进行的。

使用[键盘接口](https://www.w3.org/WAI/WCAG22/Understanding/#dfn-keyboard-interface)在定义用于前进焦点的顺序（从一个元素到下一个元素）中进行导航

用户代理 (user agent)

为用户检索和呈现网页内容的任何软件

网页 (web page)

使用 HTTP 从单个 URI 获取的非嵌套资源，以及由[用户代理](https://www.w3.org/WAI/WCAG22/Understanding/#dfn-user-agent)一起呈现或旨在一起呈现的任何其他资源

注意 1

尽管任何“其他资源”会与主要资源一起呈现，但它们不一定会同时呈现。

注意 2

为了符合这些指南，资源在一致性范围内必须是“非嵌套的”才能被视为网页。
