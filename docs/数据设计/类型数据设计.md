很好，这一步你已经在做“内容系统建模”了，而不是简单分类。

你现在的约束是：

内容类型只有三种：

1. 个人原创
2. 知识文章（外部整理）
3. 引用（摘录/授权片段）

我帮你把它们分别设计成**可落地的数据结构（适合数据库 / CMS / Notion / Supabase）**。

---

# **一、统一基础结构（所有类型共用）**

先给一个“父结构”，避免三类重复设计：

```ts
ContentBase {
  id: string
  title: string
  slug: string

  domain: string        // behavior / ux / growth / design-system / thinking / ai-product / org
  tags: string[]        // 核心标签体系

  type: "original" | "knowledge" | "quote"

  summary?: string
  created_at: Date
  updated_at: Date

  visibility: "public" | "private" | "draft"
}
```

---

# **二、类型 1：个人原创（Original）**

## **✔️ 定位**

你的观点、方法论、设计体系输出

特点：

- 有“立场”
- 有“结构化思考”
- 是知识资产核心

---

## **✔️ 数据结构**

```ts
OriginalContent extends ContentBase {
  type: "original"

  content: {
    outline?: string[]     // 文章结构（可选）
    body: string           // 正文（markdown）
  }

  idea_source?: string[]   // 灵感来源（可选）

  thinking_type?: 
    | "framework"
    | "opinion"
    | "method"
    | "analysis"
    | "case_reflection"

  related_concepts?: string[]  // 概念链接

  related_cases?: string[]      // 案例引用

  conclusion?: string
}
```

---

## **✔️ 示例**

```text
type: original

title: 如何判断一个组件是否包含业务语义

thinking_type: framework

domain: design-system

tags:
  - component
  - semantic-design
  - architecture

body:
  - 什么是业务语义
  - 为什么会污染组件层
  - 判断标准
  - 决策模型
```

---

# **三、类型 2：知识文章（Knowledge / 外部整理）**

## **✔️ 定位**

对外部内容的结构化整理，而不是复制

特点：

- 来源明确
- 结构重组
- 信息压缩
- 不表达强个人观点

---

## **✔️ 数据结构**

```ts
KnowledgeContent extends ContentBase {
  type: "knowledge"

  source: {
    title: string
    author?: string
    url: string
    publisher?: string
    published_at?: Date
  }

  content: {
    key_points: string[]      // 核心要点（重写，不是摘录）
    structure?: string[]      // 原文逻辑结构
    summary: string
  }

  interpretation?: {
    applicability?: string[]   // 适用场景
    limitations?: string[]     // 局限性
  }

  related_concepts?: string[]
  related_patterns?: string[]
}
```

---

## **✔️ 示例**

```text
type: knowledge

title: NNGroup - Form Error Design Guidelines

source:
  title: Error Message Guidelines
  url: https://...

content:
  key_points:
    - 错误提示必须靠近输入字段
    - 避免仅用颜色表达错误
    - 错误信息要可操作

  summary: 表单错误设计强调“可见性 + 可操作性 + 即时反馈”

interpretation:
  applicability:
    - SaaS 表单设计
    - 注册流程
```

---

# **四、类型 3：引用（Quote / 摘录）**

## **✔️ 定位**

精确保存外部原文片段（短引用 / 授权 / 重点摘录）

特点：

- 最接近原文
- 不重写
- 不结构化改造
- 严格控制长度

---

## **✔️ 数据结构**

```ts
QuoteContent extends ContentBase {
  type: "quote"

  source: {
    title: string
    author?: string
    url?: string
    publisher?: string
  }

  quote: {
    text: string          // 原文摘录（短）
    context?: string      // 上下文说明
    page_or_section?: string
  }

  annotation?: {
    why_important?: string   // 为什么摘录它（这是唯一允许的“解释”）
    usage_note?: string      // 用于什么设计场景
  }

  related_concepts?: string[]
}
```

---

## **✔️ 示例**

```text
type: quote

title: Don Norman on Feedback

source:
  title: The Design of Everyday Things

quote:
  text: "Feedback is essential to ensure that users understand the results of their actions."

annotation:
  why_important: 反馈是可用性的基础原则之一
  usage_note: 用于按钮点击、表单提交、系统状态设计
```

---

# **五、三种类型的核心差异（非常重要）**

## **✔️ 一张对比表**

|**类型**|**核心能力**|**是否允许重写**|**是否允许观点**|**价值来源**|
|---|---|---|---|---|
|original|创造知识|✔️|✔️ 强|你|
|knowledge|结构化外部知识|✔️|⚠️弱|外部 + 你重组|
|quote|保存原文|❌|❌极弱|原作者|

---

# **六、关键设计原则（避免未来系统崩）**

## **✔️ 原则 1：type 不表达语义，只表达“来源形态”**

```text
type = 内容来源 + 处理方式
不是知识分类
```

---

## **✔️ 原则 2：语义必须全部交给 domain + tags**

```text
domain + tags = 知识结构
type = 展示与来源结构
```

---

## **✔️ 原则 3：quote 必须“可追溯”**

```text
quote 必须包含 source + context
否则会变成版权风险点
```

---

# **七、推荐最终数据库模型（完整版）**

```ts
Content {
  id
  title
  slug

  type: original | knowledge | quote

  domain: string
  tags: string[]

  summary

  // type-specific
  original?: OriginalContent
  knowledge?: KnowledgeContent
  quote?: QuoteContent

  created_at
  updated_at
}
```

---

# **八、一个非常关键的设计建议（帮你避免后期返工）**

你现在这个系统，未来会变大，所以建议你：

## **✔️ 不要让 type 承担任何“语义判断”**

否则会变成：

- case？
- article？
- note？
- pattern？
- guide？

最后会乱掉

---

# **九、总结一句话**

你这个系统的正确结构是：

```text
domain + tags = 知识地图
type = 内容形态（来源方式）
```
