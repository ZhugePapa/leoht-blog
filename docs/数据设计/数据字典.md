# Leoht 数据字典 v1.1

> 本文档定义 Leoht 知识系统中所有数据对象的字段、类型、约束与关系。
> 当前实现状态：Astro 静态站点 + Markdown 内容集合（无数据库）。
> 后端方案见 [后端技术方案.md](../后端技术方案.md)。

> **v1.0 → v1.1 关键变更**（依据 [类型数据设计.md](类型数据设计.md) 与 [内容信息架构.md](内容信息架构.md)）：
> 1. **类型（type）从 9 种语义类型收敛为 3 种来源形态**：`original / knowledge / quote`。语义判断全部交给 domain + tags，type 只表达"内容来源 + 处理方式"。
> 2. **领域（domain）从 9 个收敛为 7 个**：移除 `visual`（是维度，归入 ui 标签）、`case-study`（是内容用途，归入 usage 标签），`leadership` → `org`，`product` → `design-system`。
> 3. **标签分组从 4 组扩展为 6 组**：`scenario / problem / product / ui / concept / usage`，替换原 `type / stage / behavior / scene`。

---

## 0. 信息模型分层

系统分三层，各层职责严格分离：

```
1. 领域（Domain）        → 一级语义骨架（稳定，7 个）
2. 标签（Tags）         → 多维语义索引（核心，6 组）
3. 类型（Type）         → 内容来源形态（UI 层，3 种）
```

**核心原则**：
- `domain + tags` = 知识地图（语义）
- `type` = 内容来源与处理方式（不参与语义分类）
- type 不表达语义，只表达"这段内容是怎么来的、怎么处理的"

详见 [内容信息架构.md](内容信息架构.md) 与 [类型数据设计.md](类型数据设计.md)。

## 2. Card（文章 / 卡片）

> 核心内容单元。一篇文章 = 一张 Card。

### 2.1 字段定义

| 字段 | 类型 | 必填 | 默认值 | 说明 |
| --- | --- | --- | --- | --- |
| `id` | string (uuid) | 系统生成 | — | 主键 |
| `title` | string | ✅ | — | 标题 |
| `slug` | string | 可选 | 取文件名 | URL 友好标识 |
| `summary` | string | — | `""` | 一句话核心观点，用于卡片 / 列表摘要 |
| `content` | text (markdown) | ✅ | — | 正文（Markdown body） |
| `type` | enum | ✅ | — | `original` / `knowledge` / `quote`（见 §2.2） |
| `status` | enum | ✅ | `inbox` | `inbox` / `draft` / `published` / `archived` |
| `tags` | string[] | — | `[]` | 标签 slug 列表（语义全部由 tags 表达） |
| `domain` | string? | — | `null` | 领域 slug（7 选 1，可选） |
| `thinking_type` | enum? | — | `null` | 仅 `original` 用：`framework / opinion / method / analysis / case_reflection` |
| `source_author` | string? | — | `""` | 来源作者（`knowledge` / `quote` 必填） |
| `source_url` | string (url)? | — | `""` | 来源链接（`knowledge` / `quote` 必填，可追溯） |
| `source_publisher` | string? | — | `""` | 来源发布方（如 NNGroup / 书名） |
| `published_at` | date? | — | `null` | 发布日期 |
| `related` | string[] | — | `[]` | 关联 Card slug 列表（自动生成 Link） |
| `created_at` | datetime | 系统生成 | — | 创建时间 |
| `updated_at` | datetime | 系统生成 | — | 更新时间 |

### 2.2 type 枚举（来源形态）

type 只表达"内容从哪来、怎么处理"，**不表达语义**。语义交给 `domain + tags`。

| 值 | 中文名 | 核心能力 | 是否允许重写原文 | 是否允许个人观点 | 价值来源 |
| --- | --- | --- | --- | --- | --- |
| `original` | 个人原创 | 创造知识 | ✔️（自己写的） | ✔️ 强 | 你 |
| `knowledge` | 知识整理 | 结构化外部知识 | ✔️ 重组（非复制） | ⚠️ 弱 | 外部 + 你重组 |
| `quote` | 引用摘录 | 保存原文 | ❌ 严禁改动 | ❌ 极弱（仅 annotation） | 原作者 |

**类型特定字段约束**：
- `original`：可填 `thinking_type`；`source_*` 通常为空
- `knowledge`：`source_author` + `source_url` 必填；正文为结构化重组（key_points / summary），非原文复制
- `quote`：`source_author` + `source_url` 必填（版权可追溯）；正文 = 精确原文摘录，可在 `summary` 写一句 annotation

> ⚠️ **设计原则**：不要让 type 承担语义判断（如 case / principle / effect / note 全部是语义，应作为 `usage` 或 `concept` 标签）。否则类型会无限膨胀直至崩坏。

### 2.3 status 枚举

| 值 | 说明 | 对外可见 |
| --- | --- | --- |
| `inbox` | 收件箱——刚记录，未整理 | ❌ |
| `draft` | 草稿——正在写 | ❌ |
| `published` | 已发布 | ✅ |
| `archived` | 归档——不再展示 | ❌ |

### 2.4 当前 Astro Schema（content.config.ts）

```typescript
z.object({
  title: z.string(),
  slug: z.string().optional(),
  summary: z.string().default(""),
  type: z.enum(["original", "knowledge", "quote"]),
  status: z.enum(["inbox", "draft", "published", "archived"]),
  tags: z.array(z.string()).default([]),
  domain: z.string().optional(),
  thinking_type: z.enum(["framework", "opinion", "method", "analysis", "case_reflection"]).optional(),
  source_author: z.string().nullable().optional().default(""),
  source_url: z.union([z.string().url(), z.literal(""), z.null()]).optional().default(""),
  source_publisher: z.string().nullable().optional().default(""),
  published_at: z.coerce.date().optional(),
  related: z.array(z.string()).default([]),
})
```

> 当前 `content.config.ts` 仍是旧的 9 类型 enum，待按上述 schema 更新（见 [后端技术方案.md](../后端技术方案.md) §4.2 同步）。

## 3. Tag（标签）

> 扁平横向组织机制。当前隐式存在于 Card.tags 数组中，无独立表。

### 3.1 字段定义

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `id` | string (uuid) | 系统生成 | 主键 |
| `name` | string | ✅ | 标签显示名（中文） |
| `slug` | string | ✅ | URL 标识（唯一） |
| `group` | enum | ✅ | `scenario / problem / product / ui / concept / usage`（见 §3.2） |
| `description` | string? | — | 标签页展示用 |
| `usage_count` | int | 系统统计 | 被引用次数（自动计算） |
| `created_at` | datetime | 系统生成 | — |

### 3.2 标签分组（6 大类）

> 标签是本系统的**核心资产**——比领域更重要，负责跨领域连接。语义分类全部由标签承担。

| 分组 | 含义 | 作用 | 示例 |
| --- | --- | --- | --- |
| `scenario` | 场景 | 用户"按任务找知识"（使用频率最高） | onboarding / signup / pricing / checkout / form / search / empty-state / error-state / settings |
| `problem` | 问题 | 描述"设计在解决什么问题"，用于分析本质 | conversion / activation / retention / clarity / usability / trust / cognitive-load / decision-fatigue |
| `product` | 产品类型 | 区分设计环境 | b2c / b2b / saas / mobile / web / admin / ai-app / enterprise |
| `ui` | 设计对象 | 描述"设计作用在哪"（原 visual 维度归此） | layout / typography / spacing / color / icon / component / interaction / navigation / table / chart |
| `concept` | 概念 | 跨领域基础认知，建立理论体系 | mental-model / affordance / feedback / constraint / hierarchy / progressive-disclosure / fitts-law / gestalt |
| `usage` | 内容用途 | 区分"这东西是什么用途"（原 type/stage 的语义部分归此） | pattern / principle / case-insight / reference / guideline / best-practice / anti-pattern / framework |

### 3.3 标签治理规则

> 避免标签随意增长导致系统崩坏。

1. 标签必须归属某一类（scenario / problem / product / ui / concept / usage）
2. 禁止重复语义标签（如 `usability` vs `user-friendly`，二选一）
3. `scenario` 优先于 `concept`（能用场景描述就不用概念）
4. `problem` 优先用于分析
5. `ui` 仅描述界面结构，不承载语义
6. 每条内容标签数量上限：
   - `scenario` ≤ 2
   - `problem` ≤ 3
   - `ui` ≤ 5
   - `concept` / `product` / `usage` 各 ≤ 3

## 5. Domain（领域）

> Domain 是一级语义骨架——"设计问题发生在哪个系统里"。稳定、不轻易变化。
> 当前硬编码在 domains.ts，目标为数据库表。

### 5.1 字段定义

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `slug` | string | ✅ | URL 标识（主键） |
| `name` | string | ✅ | 中文名 |
| `enName` | string | ✅ | 英文名 |
| `question` | string | ✅ | 该领域要回答的核心问题 |
| `description` | string | ✅ | 领域描述 |
| `color` | string | ✅ | CSS 变量引用（如 `var(--color-charcoal)`） |
| `soft` | string | ✅ | 浅色变体（CSS 变量 / rgb） |
| `image` | string? | — | 领域配图文件名（亮色） |
| `abbr` | string | 系统生成 | 3 字母缩写（如 BEH / UX / GRO） |

### 5.2 当前定义（7 个稳定领域）

| slug | name | enName | abbr | question |
| --- | --- | --- | --- | --- |
| `behavior` | 人与行为 | Behavior | BEH | 用户为什么这样做 |
| `ux` | 体验设计 | UX | UX | 产品应该如何设计 |
| `growth` | 产品增长 | Growth | GRO | 设计如何影响业务 |
| `design-system` | 设计系统 | Design System | DS | 设计如何落地 |
| `thinking` | 方法论与思维 | Thinking | THK | 设计师如何思考 |
| `ai-product` | AI 产品设计 | AI Product | AIP | AI 如何改变交互 |
| `org` | 组织与协作 | Org | ORG | 设计如何被组织采纳 |

### 5.3 收敛说明（v1.0 → v1.1）

原 9 个领域收敛为 7 个，移除的 2 个不是领域而是维度/内容类型：

| 移除/改名 | 原因 | 归宿 |
| --- | --- | --- |
| `visual` 移除 | 是维度（视觉）不是领域 | 归入 `ui` 标签（typography / color / layout） |
| `case-study` 移除 | 是内容用途不是领域 | 归入 `usage` 标签（case-insight） |
| `leadership` → `org` | 更中性，覆盖协作+管理 | 改名 |
| `product` → `design-system` | 聚焦设计系统而非泛产品 | 改名 |

> 领域判定原则：领域 = "设计问题发生在哪个系统里"。`visual` 回答的是"形式如何承载意义"——那是设计对象的维度，应作为 `ui` 标签。`case-study` 回答的是"别人怎么做"——那是内容用途，应作为 `usage` 标签。

## 11. 编号体系

Card 有三段编号 `{global}.{domainAbbr}.{domainSeq}`：

| 段 | 含义 | 生成规则 |
| --- | --- | --- |
| `global` | 全局期号（罗马数字） | 按发布时间升序，最早 = i（`toRoman(total - i)`） |
| `domainAbbr` | 领域缩写（见 §5.2） | `DOMAIN_ABBR[domain] ?? domain.slice(0,3).toUpperCase()` |
| `domainSeq` | 域内序号 | 同领域内按时间升序递增 |

7 领域缩写对照：`BEH / UX / GRO / DS / THK / AIP / ORG`。

示例：`i.ORG.1` = 全刊第 1 篇 / 领域 org / org 域内第 1 篇。

> ⚠️ 历史内容（v1.0 的 `iii.AIP.1` / `ii.VIS.1` / `i.LDR.1`）需在迁移时重新编号：`VIS` → 归入 `DS` 或 `UX` 域；`LDR` → `ORG`。见 [后端技术方案.md](../后端技术方案.md) §6 迁移策略。
