# Leoht 后端技术方案 v1.1

> 从 Astro 静态站点到有后端的知识系统——分阶段落地计划。
> 数据对象定义见 [数据字典.md](./数据设计/数据字典.md)。
> 信息模型分层（领域 / 标签 / 类型）见 [数据设计/内容信息架构.md](./数据设计/内容信息架构.md) 与 [数据设计/类型数据设计.md](./数据设计/类型数据设计.md)。

---

## 1. 现状

| 维度 | 当前 |
| --- | --- |
| 框架 | Astro 7（静态生成 `output: "static"`） |
| 内容 | Markdown 文件 + Zod schema（`content.config.ts`） |
| 数据存储 | 文件系统（`/content` 目录，git 版本管理） |
| 搜索 | 无（内容库靠客户端 JS 过滤） |
| 认证 | 无（header 有登录按钮，无功能） |
| 订阅 | 无（newsletter 表单 onsubmit="return false"） |
| 部署 | GitHub Pages（`zhugepapa.github.io/leoht-blog`） |

**优势**：零运维成本、git 即版本管理、Markdown 即 CMS、构建速度快。

**瓶颈**：无法动态管理标签 / 无法实时收订阅 / 无搜索 / 无评论 / 无多设备管理。

## 2. 分阶段目标

| Phase | 目标 | 关键能力 | 触发条件 |
| --- | --- | --- | --- |
| **P0 — 当前** | 静态知识库 | Markdown 内容 + 客户端筛选 + 暗色模式 | 已完成 |

## 3. P1 — 轻量后端

### 3.1 技术选型

| 能力 | 方案 | 理由 |
| --- | --- | --- |
| 搜索 | **Pagefind**（静态索引） | 无后端、构建时生成索引、纯前端搜索 |
| 订阅 | **Resend** + **Astro API Route** | Resend 免费额度 100/天；Astro endpoint 接收 POST |
| 标签聚合 | **Astro 动态路由** `/tag/[slug]` | 构建时生成标签页，无需数据库 |
| RSS | **@astrojs/rss** | 官方集成，构建时生成 |
| 数据存储 | **不引入数据库** | 订阅用 JSON 文件 / Google Sheet 存储 |

### 3.2 搜索实现（Pagefind）

```
# 构建后运行 Pagefind 索引
npx pagefind --site dist
```

- 构建产物 `dist/` → Pagefind 索引 → `dist/pagefind/`
- 前端引入 Pagefind UI，搜索框在 header
- 支持中文分词（Pagefind 内置 CJK 支持）

### 3.3 订阅实现（Resend + API Route）

```
src/pages/api/subscribe.ts
```

```typescript
import type { APIRoute } from "astro";

export const POST: APIRoute = async ({ request }) => {
  const email = (await request.formData()).get("email");
  // 1. 验证 email
  // 2. 写入存储（JSON 文件 / Google Sheet / Supabase）
  // 3. 返回 { success: true }
  return new Response(JSON.stringify({ success: true }), {
    headers: { "Content-Type": "application/json" },
  });
};
```

**存储方案**：初期用 `data/subscribers.json`（git 管理），后期迁移到 Supabase。

### 3.4 标签聚合页

```
src/pages/tag/[slug].astro
```

- `getStaticPaths()` 从所有 Card 的 tags 生成路径
- 每个标签页展示：标签名 / 描述 / 所属分组 / 内容数量 / 该标签下所有 Card / 相关标签

## 4. P2 — 内容管理

### 4.1 技术选型

| 能力 | 方案 | 理由 |
| --- | --- | --- |
| 数据库 | **Turso (SQLite)** 或 **Supabase (Postgres)** | Turso 免费 9GB / Supabase 免费 500MB |
| ORM | **Drizzle ORM** | 轻量、类型安全、SQLite / Postgres 双支持 |
| 认证 | **Clerk** 或 **Auth.js** | Clerk 开箱即用 / Auth.js 自托管 |
| API | **Astro Server Mode** + **API Routes** | 切换 `output: "server"` |
| 迁移 | **Drizzle Kit** | schema → migration SQL |

### 4.2 数据库 Schema（Drizzle）

```typescript
// src/db/schema.ts
import { sqliteTable, text, integer } from "drizzle-orm/sqlite-core";
import { sql } from "drizzle-orm";

export const cards = sqliteTable("cards", {
  id: text("id").primaryKey().$default(() => crypto.randomUUID()),
  title: text("title").notNull(),
  slug: text("slug").notNull().unique(),
  summary: text("summary").default(""),
  content: text("content").notNull(),
  // type = 来源形态，不参与语义分类。语义交给 domain + tags
  type: text("type", { enum: ["original", "knowledge", "quote"] }).notNull(),
  status: text("status", { enum: ["inbox", "draft", "published", "archived"] }).notNull().default("inbox"),
  domain: text("domain"),  // 7 个稳定领域之一，可选
  thinkingType: text("thinking_type", { enum: ["framework", "opinion", "method", "analysis", "case_reflection"] }),
  sourceAuthor: text("source_author").default(""),
  sourceUrl: text("source_url").default(""),            // knowledge / quote 必填（可追溯）
  sourcePublisher: text("source_publisher").default(""),
  publishedAt: integer("published_at", { mode: "timestamp" }),
  createdAt: integer("created_at", { mode: "timestamp" }).$default(() => new Date()),
  updatedAt: integer("updated_at", { mode: "timestamp" }).$default(() => new Date()),
});

export const tags = sqliteTable("tags", {
  id: text("id").primaryKey().$default(() => crypto.randomUUID()),
  name: text("name").notNull(),
  slug: text("slug").notNull().unique(),
  // 6 组语义标签，替换原 type/stage/behavior/scene
  group: text("group", { enum: ["scenario", "problem", "product", "ui", "concept", "usage"] }).notNull(),
  description: text("description"),
  usageCount: integer("usage_count").default(0),
  createdAt: integer("created_at", { mode: "timestamp" }).$default(() => new Date()),
});

export const cardTags = sqliteTable("card_tags", {
  cardId: text("card_id").notNull().references(() => cards.id),
  tagId: text("tag_id").notNull().references(() => tags.id),
});

export const links = sqliteTable("links", {
  id: text("id").primaryKey().$default(() => crypto.randomUUID()),
  sourceCardId: text("source_card_id").notNull().references(() => cards.id),
  targetCardId: text("target_card_id").notNull().references(() => cards.id),
  relationType: text("relation_type"),
  createdAt: integer("created_at", { mode: "timestamp" }).$default(() => new Date()),
});

export const domains = sqliteTable("domains", {
  slug: text("slug").primaryKey(),
  name: text("name").notNull(),
  enName: text("en_name").notNull(),
  question: text("question").notNull(),
  description: text("description").notNull(),
  color: text("color").notNull(),
  soft: text("soft").notNull(),
  image: text("image"),
});

export const subscriptions = sqliteTable("subscriptions", {
  id: text("id").primaryKey().$default(() => crypto.randomUUID()),
  email: text("email").notNull().unique(),
  status: text("status", { enum: ["active", "unsubscribed"] }).notNull().default("active"),
  token: text("token").notNull(),
  subscribedAt: integer("subscribed_at", { mode: "timestamp" }).$default(() => new Date()),
  unsubscribedAt: integer("unsubscribed_at", { mode: "timestamp" }),
});

export const collections = sqliteTable("collections", {
  id: text("id").primaryKey().$default(() => crypto.randomUUID()),
  title: text("title").notNull(),
  slug: text("slug").notNull().unique(),
  description: text("description"),
  domain: text("domain"),
  cardIds: text("card_ids", { mode: "json" }).$type<string[]>().notNull(),
  order: integer("order").default(0),
  createdAt: integer("created_at", { mode: "timestamp" }).$default(() => new Date()),
});
```

### 4.3 API 设计

```
# Card CRUD
GET    /api/cards              → 列表（支持 ?domain= &tag= &type= &status= &page= &limit= ）
GET    /api/cards/:slug        → 详情
POST   /api/cards              → 创建（需 auth）
PUT    /api/cards/:slug        → 更新（需 auth）
DELETE /api/cards/:slug        → 删除（需 auth）

# Tag
GET    /api/tags               → 全部标签（含 usage_count）
GET    /api/tags/:slug         → 标签详情 + 该标签下 Cards
POST   /api/tags               → 创建（需 auth）
PUT    /api/tags/:slug         → 更新 / 合并（需 auth）

# Domain
GET    /api/domains            → 全部领域
PUT    /api/domains/:slug      → 更新（需 auth）

# Subscription
POST   /api/subscribe          → 订阅
POST   /api/unsubscribe        → 退订（带 token）

# Search
GET    /api/search?q=          → 全文搜索（P2 可用 Meilisearch 替代 Pagefind）

# Auth
POST   /api/auth/login         → 登录
POST   /api/auth/logout        → 登出
GET    /api/auth/me            → 当前用户
```

### 4.4 Astro Server Mode 切换

```javascript
  output: "server",       // 从 "static" 切换

```

部分页面仍可静态生成（`export const prerender = true`），只有 API routes 动态执行。

## 6. 迁移策略

### 6.1 P0 → P1（无需迁移）

- Pagefind 在构建后索引 `dist/`，不改动内容
- 订阅 API 是新 endpoint，不影响现有页面
- 标签聚合页是新的 `[slug].astro` 路由

### 6.2 P1 → P2（Markdown → 数据库）

```
# 一次性迁移脚本
src/scripts/migrate-to-db.ts
```

- 读取 `content/**/*.md` 的 frontmatter + body
- 插入 cards 表（slug / title / content / type / status / ...）
- 从 tags 数组提取去重，插入 tags + cardTags
- 从 related 数组，按 slug 查 target id，插入 links
- 迁移后保留 Markdown 文件作为备份（双写过渡期）

**v1.0 → v1.1 语义迁移**（迁移脚本需处理）：

| 旧值（v1.0） | 新值（v1.1） | 处理 |
| --- | --- | --- |
| `type: article / note / highlight / principle / effect / case / retro / reading / spark` | `type: original / knowledge / quote` | 按来源形态映射：原创类（article/note/spark/retro）→ `original`；摘录类（highlight/reading/effect）→ `quote`；整理类 → `knowledge`。原语义迁移为 `usage` / `concept` 标签 |
| `domain: visual` | 移除 | 内容归入 `design-system` 或 `ux`，标签补 `ui`（typography/color/layout） |
| `domain: case-study` | 移除 | 内容归入对应领域，标签补 `usage: case-insight` |
| `domain: leadership` | `org` | 直接改名 |
| `domain: product` | `design-system` | 直接改名 |
| tag group `type/stage/behavior/scene` | `scenario/problem/product/ui/concept/usage` | 按标签语义重新归类（见数据字典 §3.2） |

**双写策略**：P2 过渡期，Markdown 和数据库双写。构建时优先读数据库，Markdown 作 fallback。稳定后删 Markdown。

### 6.3 数据备份

- Turso: 自动每日备份（免费计划）
- Supabase: pg_dump 定时任务
- Markdown: git 即版本备份（永久保留）

## 7. 安全方案

| 威胁 | 措施 |
| --- | --- |
| 未授权写入 | API 写操作验证 JWT（Clerk / Auth.js） |
| XSS | 正文 Markdown 渲染前 sanitize（DOMPurify） |
| CSRF | API endpoint 验证 Origin header |
| 订阅注入 | email 验证 + rate limit（5 次/分钟/IP） |
| SQL 注入 | Drizzle ORM 参数化查询 |
| 数据丢失 | Turso 自动备份 + git 保留 Markdown |
